# 同步

写这篇札记是因为在开发过程中反复遇到同一个问题：AI 生成的代码和文档经常没有被及时提交，或者提交了没有推送，或者推送了但父仓库的子模块指针没有更新。

每次都要额外提醒"记得提交"、"记得 push"、"记得更新父仓库引用"。这不是 AI 的能力问题，而是工具设计问题——工具把记忆责任推给了人（或 AI），但人（和 AI）的记忆是不可靠的。

这让我开始想一个问题：为什么"同步"这件事不能是工具自动完成的？

## 软约束为什么不够

目前我们依赖的"最佳实践"本质上都是软约束：

- AGENTS.md 里写明"每次修改后提交并推送"，但 AI 在长对话中会遗忘这个约定
- 对话中口头约定"做完这一步就提交"，但上下文切换后就不记得了
- 开发者自己也会忘记，尤其是在高频迭代的节奏中

这些约束的共同特征是：**需要有人记住去做**。而高频场景下，记忆恰恰是最不可靠的环节。

这和人的态度无关。不是不重视，是每次操作后多出"提交"这一步，在惯性中就容易被跳过。

## 刚性的思路

有没有可能让"同步"不是一个人为步骤，而是工具内置的行为？

我想到的几条：

**操作即提交** — 每次原子操作完成后，自动 stage 并 commit 子模块变更到父仓库。不需要用户或 AI 额外执行 `git add && git commit`。

**提交即推送** — 有网络时，commit 后自动 push。网络不可达时，标记延迟推送队列，稍后重试。

**延迟告警** — 未推送的 commit 累计超过阈值（3 个或 30 分钟），阻塞后续操作并提示推送。不是靠人记住，是靠系统阻止人继续。

**CI 验证** — 父仓库的 CI 流水线自动检查子模块指针是否已推送到远程，未推送的标记为失败。

## 工具应该承担记忆责任

这几条的核心逻辑是一样的：工具能自动做的事情，不应该让人记住去做。

每一步可追溯，每一版可回退。工具负责执行和记录，人只负责决策。这个分工清晰之后，"忘记提交"的问题就不再是问题了。
