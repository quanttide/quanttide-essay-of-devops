# 预发布版本

## 使用条件

昨天发现 CI 反复失败——发布流程涉及多步自动化（构建、测试、manylinux 校验、PyPI 上传），每一步都可能挂。挂了就得修、重打 tag、重来一轮。

这种场景下，预发布版本才有意义：

- CI 跑完整流程，但产物不标记为正式版
- 确认全部通过后，再正式发布同一个版本
- 版本号一旦发布就不可变，CI 失败的版本不会污染正式发布序列

手动发布时，预发布版本意义不大。一个人盯着终端，几步操作几秒完成，失败了回滚标签重新来就是。不需要预发布这个中间态。

## CHANGELOG 规范

CHANGELOG 中不写版本更新内容，而是注明本次预发布的目的。

```
## [v0.2.0-rc.1] — 2026-05-24

### Pre-release

- 验证 CI manylinux 构建流程
- 验证 PyPI 发布权限

## [v0.2.0-rc.2] — 2026-05-24

### Pre-release

- 修复 auditwheel 平台标签问题
- 添加 uv 安装步骤到 CI
```

正式发布时，CHANGELOG 应有独立的、完整的变更描述，不引用预发布版本：

```
## [v0.2.0] — 2026-05-25

### Added

- 新增 code status 命令
- 新增 code sync 命令

### Fixed

- 修复 PyPI 发布时 manylinux 兼容性问题
```

如果预发布版本把真正的变更内容写进 CHANGELOG，到正式发布时就没什么可写的了——要么重复，要么空白。把预发布版本的目的和正式版的变更分开写，各司其职。

## 版本号不可变规范

版本号一旦发布，无论成功失败，都不再复用。

- rc.1 失败了 → 用 rc.2 重试
- rc.2 也失败 → 用 rc.3 重试
- rc.N 成功 → 直接升为正式版 v0.2.0

失败的不删除、不覆盖、不回收。每次失败都是一次记录，保留审计轨迹。

充分利用预发布版本的**不可变性**：版本号是永恒的标签，不是可擦写的草稿。
